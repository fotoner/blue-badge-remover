# 신뢰성

## 에러 처리 정책

### 공통 원칙 (불변)

- **외부 API 호출**: 최대 **3회 재시도** (지수 백오프: 1초, 2초, 4초). 재시도 실패 시 에러를 로깅하고 해당 작업을 스킵
- **DB 쿼리 실패**: 에러 메시지를 사용자에게 반환. 배치 작업은 개별 행이 아닌 배치 단위로 에러 처리
- **파일/데이터 파싱**: 개별 레코드 파싱 실패 시 해당 레코드를 스킵하고 경고 로그. 전체 파일 실패 시 에러 기록하고 작업 중단
- **사용자 입력 검증 실패**: 에러 메시지를 사용자에게 반환

### 프로젝트 특화 정책

### X(Twitter) API
- 팔로우 목록 조회 실패: 최대 3회 재시도 (지수 백오프). 실패 시 기존 캐시 데이터 유지
- 토큰 만료/무효: 사용자에게 재인증 안내 메시지 표시. 자동 재시도 금지
- Rate limit (429): Retry-After 헤더 기반 대기 후 재시도

### Chrome Storage API
- 저장 실패: 에러 로그 기록, 사용자에게 설정 저장 실패 알림
- 용량 초과: 오래된 캐시 데이터부터 정리 후 재시도

### DOM 조작
- 셀렉터 매칭 실패 (X UI 변경): 경고 로그 기록, 해당 요소 스킵. 익스텐션 전체가 중단되지 않아야 함

## 데이터 무결성

### 공통 원칙 (불변)

- **감사 추적**: 데이터 변경 작업(임포트, 마이그레이션 등)은 로그 테이블에 기록 (소스, 시작/완료 시각, 상태, 처리 건수, 에러)
- **중복 방지**: 핵심 엔티티에 UNIQUE 제약 적용. 삽입은 upsert(ON CONFLICT) 패턴 사용
- **출처 추적**: 외부 데이터에서 파생된 값에는 source와 confidence/quality 필드를 기록

## 로깅

- **형식**: 구조화된 로그 (JSON 권장) `{ timestamp, level, message, ...data }`
- **레벨**: debug, info, warn, error (기본: info, 환경 변수로 변경 가능)
- **원칙**: 로그에 민감 정보(API 키, 비밀번호, 개인정보) 출력 금지
